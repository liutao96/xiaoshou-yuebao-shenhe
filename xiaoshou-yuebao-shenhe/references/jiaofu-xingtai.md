# 最终交付形态规则

## 总原则

月报审核结果不能只停留在几句聊天文字。正式初审、深度诊断、C/D 打回整改，默认要形成可留档的审核报告。

## 默认交付

### 用户只发飞书链接

默认输出三层，且完整报告必须双交付：

1. 聊天区部长首屏回复：给出读取状态、内部审计分流结果、评分、是否可提交老板、最大问题、14 模块速览、部长下一步动作。
2. 完整 14 模块主报告：默认生成 Word 正式版 + Markdown 底稿；如果当前环境只能生成 Markdown，必须说明 Word 未生成的原因。B/C/D 的附加交付物默认合并在主报告前部，作为“部长重点看”章节。
3. 最后一段可转发整改指令：给主管或部门直接照着改，不能只写“见报告”。

部长或管理者只贴飞书链接时，默认就是“部长要看初审结果”。不能只输出简单摘要，必须先在聊天区给 14 模块速览，再给 Word 正式版和 Markdown 底稿路径，并明确说明部长重点看主报告哪一节。

### 完整 14 模块审核

必须生成完整 Word 正式版和 Markdown 底稿。聊天区不用把全部长文重复一遍，但必须按 `buchang-shouping-huifu.md` 给每个模块至少一句速览，不能只放关键结论和报告路径。

完整报告前部必须加入“本报告生成逻辑”，说明读取范围、审核路径、14 模块框架、评分权重、最高分限制和 A/B/C/D 分流规则。

完整报告前部还必须按分流结果加入“部长重点看”章节：

- A 类：主报告前部写“老板会可看要点”，并额外单独生成老板一页纸正式版。
- B 类：主报告前部写“补充小问题清单 / 部长重点看”。
- C 类：主报告前部写“主管整改版 / 部长重点看”。
- D 类：主报告前部写“重做通知 / 部长重点看”。

这些重点章节必须放在 14 模块正文之前，方便部长打开一个主报告就能先看到处理动作。

### C 类或 D 类

必须在主报告前部生成“主管整改版 / 部长重点看”章节或“重做通知 / 部长重点看”章节。聊天区展示分流、分数、真实存在的 1-5 个整改问题和提交要求；完整报告保留口径说明、费用整改、品项整改、动作闭环、老板追问、复审标准。桥接表和复算表仅在重大冲突时加入。C/D 默认不另拆主管整改包文件，除非用户明确要求单独文件。

### A 类或 B 类

A 类默认额外单独生成老板一页纸正式版，便于直接给老板或会议使用；同时仍保留完整 14 模块主报告。

B 类默认不另拆文件，在主报告前部生成“补充小问题清单 / 部长重点看”章节；聊天区必须说明补充完即可提交老板。若用户明确要求单独文件，再单独生成轻量补充清单或老板一页纸草稿。

### 复审

聊天区输出复审结论、是否建议提交老板、最多 5 条剩余整改项。复审通过并需要留档时，生成老板一页纸 Word 正式版和 Markdown 底稿。

## Markdown 与 Word 的选择

| 场景 | 首选形态 | 说明 |
|---|---|---|
| Codex/AI 内部审核 | Markdown + Word | Markdown 便于追踪，Word 便于业务人员打开 |
| 部长贴链接快速看结果 | 聊天区 14 模块速览 + 主报告 Word 正式版 + Markdown 底稿 + 可转发整改指令 | 不打开文档也能知道各模块审出了什么；打开主报告先看“部长重点看”章节 |
| A 类给老板月会 | 单独老板一页纸 Word + 主报告 Word/Markdown | 老板一页纸干净独立，主报告留档 |
| B/C/D 给主管整改 | 主报告 Word + Markdown，重点区放前部 | 不默认拆多个文件，减少部长找文件成本 |
| 飞书文档沉淀 | Markdown 正文或飞书 Docx | 如果有写入权限，可创建飞书文档 |

## 文件命名

生成本地报告时使用：

```text
outputs/feishu_monthly_audit/YYYY-MM_部门或小组_销售月报AI初审报告.md
outputs/feishu_monthly_audit/YYYY-MM_部门或小组_销售月报AI初审报告.docx
outputs/feishu_monthly_audit/YYYY-MM_部门或小组_老板一页纸正式版.docx
outputs/feishu_monthly_audit/YYYY-MM_部门或小组_销售月报深度经营诊断.md
outputs/feishu_monthly_audit/YYYY-MM_部门或小组_销售月报深度经营诊断.docx
outputs/feishu_monthly_audit/YYYY-MM_部门或小组_销售月报复审结果.md
outputs/feishu_monthly_audit/YYYY-MM_部门或小组_销售月报复审结果.docx
```

如果无法识别部门或月份，用文档标题和当前日期代替。

## Word 文档生成规则

以下情况必须默认生成 Word：

- 正式初审；
- 部长或管理者只贴链接发起审核；
- 完整 14 模块报告；
- A 类老板一页纸正式版；
- 深度经营诊断；
- 复审结果需要留档；
- 用户明确说要 Word、docx、正式发老板、正式发主管；
- 当前环境有可用文档处理能力；
- 报告已经先形成 Markdown 底稿。

Word 生成失败时，不中断审核结论，交付 Markdown 作为正式底稿，并说明未能生成 Word 的原因。

## 不允许

- 不允许只输出一句“建议打回整改”。
- 不允许把完整审核只放在心里，最终不给用户可追溯材料。
- 不允许为了正式感生成 Word，但内容仍然是简化结论。
- 不允许正式初审只生成 `.md` 而不尝试生成 Word。
- 不允许业务人员场景只给 Markdown 文件。
- 不允许给部长的初审报告缺少“本报告生成逻辑”。
- 不允许 B/C/D 主报告缺少“部长重点看”章节。
- 不允许部长首屏回复只写 3 条硬伤和一个报告路径。
- 不允许用“详见报告”替代 14 模块速览。

## 最后一段可转发整改指令

每次正式初审结束后，聊天区最后必须给一段可直接复制转发给主管/部门的文字。

要求：

- 不写成 AI 自我说明；
- 不写复杂提示词；
- 语气像部长布置整改；
- 明确是否需要重写全文，还是补充关键说明；
- 优先说明原月报需要改哪些备注、口径和动作；只有必要时才要求补表；
- 明确补完后提交复审。

模板：

```text
本次 AI 初审结论：建议先补充整改后再提交老板，不需要重写全文。请优先在原月报中补充本次初审明确指出的内容：主销售口径及关键差异说明、问题的真实归因、重点品项/渠道/客户的具体改进动作、下月计划的负责人和完成时间。只有月报已出现费用率、ROI、预算费用或明确费用结论时，才补费用率分母和异常原因；只有初审报告明确标为“条件必须”的材料才需要补表。修改完成后，直接把新版月报链接和本次 AI 初审报告一起提交复审。
```

如果报告为 A/B 类，可改为：

```text
本次 AI 初审结论：整体可以进入老板审阅，但提交前请补齐口径说明和关键差异解释，避免老板追问时无法当场回答。请按初审报告中的待补清单补充后，再提交最终版。
```
