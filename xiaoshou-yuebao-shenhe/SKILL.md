# 销售月报审核

## 定位

把销售小组月报按“初审刀、复审刀、深度刀”机制做内部审计：先读原始月报和关键表格，再判断材料完整性，再按 14 模块完成内部深审，最后评分、A/B/C/D 分流，并按场景交付聊天结论、Markdown 报告或 Word 文档。

这不是普通总结、润色或绩效评价。评分性质始终是“月报质量与经营判断支撑度评分”，不是部门绩效分。

正式审核不能只输出几句简单结论。即使最终为 C/D，也必须先完成数据、经营、渠道、品项、费用、漏斗、动作闭环、问题归因、管理风险、下月建议、老板追问和评分等内部判断。

## 月报内提示文字的处理

正式月报中保留的“填写说明”“AI审核点”“格式合规提醒”“模板说明”“示例/占位提示”等内容，属于给查看者看的辅助提醒，不是业务事实，也不是审核错误。

审核时必须做到：

1. 不要求业务人员删除这些提醒，不把它们列为主要硬伤、整改项或复审通过条件。
2. 不因这些提醒存在而扣分、触发最高分限制或判定月报不合规；它们可以继续保留在正式月报中。
3. 正式报告可单独列出“已识别的查看提醒”，说明这些内容已保留、仅作阅读辅助，本次不计错。
4. “明显模板错误”只指错误月份、上月数字未替换、字段内容明显复制错误、实际数据互相冲突等已经影响本月内容的事实问题，不包括提示文字本身。

## 资产边界

本 Skill 不内置月报填报模板或条件化补充工具包。审核只依赖用户提交的月报原文、关键表格和 `references` 规则；正式审核报告仍按 `word-jiaofu-guize.md` 生成 Word + Markdown。

## 入口判断

先判断用户给的材料类型：

1. 用户只发一条飞书链接：直接按销售月报审核处理，不要求用户再写提示词。
2. 飞书链接：先用飞书/Lark 相关能力读取原文和表格。通常按链接类型使用 `lark-drive`、`lark-doc`、`lark-sheets`、`lark-wiki`，认证和权限问题走 `lark-shared`。
3. 本地或上传文件：读取 Word、Excel、PDF 或文本内容。表格必须尽量保留行列结构。
4. 纯文本粘贴：可以试审，但要标注“仅基于用户粘贴文本，无法核验原表格和附件”。
5. 复审材料包或简单复审入口：如果包含“整改后、修改后、复审、复核、重新提交、已按意见修改”等信息，优先按复审处理，不要当首次初审。即使用户只发整改后月报链接，也先自动检索上次 AI 初审结果；找不到时再让用户补充。
6. 重做材料包：如果包含“重做后月报、上次重做通知、重做对照说明表”，优先按重做复核处理。

再判断使用哪把刀：

- 只发链接、首次提交、普通“审核这个月报”：默认初审刀。
- 出现复审、复核、整改后、修改后、重新提交、已按意见修改、对照上次 AI 结果：使用复审刀；先自动找上次初审依据，找不到再提示补充。
- 出现深度审核、经营诊断、老板会、14 模块、不要简单回答、重点部门：使用深度刀。
- 只要求口径、费用率、数据冲突：使用专项检查，不强行输出完整分流。

## 环境自检与授权补齐

在新电脑、部长电脑或不确定环境中，收到飞书链接后先做自检，不要先要求用户解释怎么用：

1. 检查是否有 `lark-cli`。Windows 可用 `Get-Command lark-cli -ErrorAction SilentlyContinue`；其他系统用 `command -v lark-cli`。
2. 如果没有 `lark-cli`，由 agent 先尝试按当前环境可用方式安装或恢复飞书 CLI；如果当前环境没有安装器、没有网络或没有管理员权限，明确说明缺少什么，并只问一个最关键问题，不继续假装能读取飞书。
3. 如果有 `lark-cli` 但未初始化，先按 `lark-shared` 执行 `lark-cli config init --new`。命令输出授权链接时，必须生成二维码并展示给用户完成配置。
4. 如果已初始化但未登录或权限不足，按 `lark-shared` 发起 user 身份授权。月报读取优先授权这些域：`drive,docs,sheets,wiki,base`。如果 CLI 返回具体缺失 scope，优先按缺失 scope 增量授权。
5. 授权必须用 split-flow：先执行 `lark-cli auth login --domain drive,docs,sheets,wiki,base --no-wait --json` 或按缺失 scope 登录，生成二维码和链接给用户；等用户回复已授权，再由 agent 执行 `lark-cli auth login --device-code <device_code>` 完成登录。
6. 不要求部长手动运行命令。除扫码/点击授权这类必须由本人完成的动作外，安装、自检、配置、读取、重试都由 agent 推进。

授权和安装没有完成前，不能生成正式审核结论。最多输出“环境未就绪/权限未就绪，暂无法读取飞书原文”。

### 多应用 / 多 profile 权限探测

销售月报审核运行中，如果公司默认 profile 没有权限，不要立刻判定“飞书原文无法读取”。先确认当前链接属于公司内容，再在公司 profile 范围内做只读排查。

执行规则：

1. 默认使用 `company-liutao-default`，先做只读盘点：`lark-cli config show`、`lark-cli auth list --json`；输出中出现 appSecret、token、cookie、secret 等必须脱敏。
2. 只在公司 profile 范围内尝试读取。不得为了通过权限自动切到 `personal-do-not-use-company-content` 或 `outsource-approval-report-sync-bot-no-user`；`company-jiqian-meeting-archive` 仅在任务明确涉及纪倩时使用。
3. 如果默认走到 bot 且 bot 看不到用户文档，优先在 `company-liutao-default` 发起 user 身份授权或请文档所有者授权，不把“能访问的其他环境”当成可用环境。
4. 找到可访问的公司 profile 后停止继续尝试；正式报告只说明已读取原文和关键表格，不暴露 appId、token、secret。
5. 如果公司 profile 仍失败，再判断是文档未授权、scope 不足、账号不对，还是需要用户重新授权或让文档所有者开权限。

边界：

- 这条多 profile 探测只适用于读取月报原文、表格、附件信息等只读操作。
- 写入、覆盖、删除、同步、移动文件、发消息、审批、修改权限、创建文档等有副作用操作，不能自动跨 profile 尝试，必须先得到用户明确确认。
- 如果用户明确指出某个 profile / 应用是当前电脑的错误配置或不该用于本任务，就跳过该 profile；但不要把单台电脑的问题写成所有电脑通用禁令。
- 多 profile 探测是为了降低月报读取误判率，不能用来绕过权限、冒用身份或扩大写入范围。

## 飞书链接读取规则

用户只给飞书链接时，先做只读解析：

1. 识别链接属于 Docx、Sheet、Wiki、Drive 文件、Base 或嵌入资源。
2. 读取正文、标题、表格、附件信息和可见的嵌入表格。
3. 如果链接是 Wiki 节点，先解析真实对象类型，再转对应读取路径。
4. 如果文档里有附件、图片表格、嵌入电子表格但无法读取，必须写入“材料完整性检查”，不能直接当作完整月报正式评分。
5. 如果权限不足、登录态缺失、链接不可访问，明确说明“无法读取飞书原文，不能生成正式月报审核结论”，并列出需要谁授权或补传什么。

不要把“成功打开标题”当作“成功读取月报”。月报审核必须基于正文和关键表格。

## 材料完整性检查

正式评分前先输出或内部确认这些项：

| 检查项 | 要求 |
|---|---|
| 部门和月份 | 能识别部门、小组、月份 |
| 主销售口径 | 能看出当前主口径是什么 |
| 关键表格 | 销售、渠道、费用、品项、动作闭环等核心表格尽量可读 |
| 附件和嵌入表格 | 已读取或明确标注无法读取 |
| 复审材料 | 必须有整改后月报；优先自动找到上次 AI 初审结果，找不到才请用户补；整改回复为可选说明，不是硬性材料 |

如果材料不足但仍可做初步判断，只输出“试审/初步审核”，不要写成正式通过。缺少关键表格、无法读取附件、主口径无法判断时，不要强行给 A。

## 审核流程

按这个顺序做：

1. 识别任务：初审、复审、重做复核、专项检查。
2. 识别小组类型：平台店铺、内容达人、新零售/分销、混合型。
3. 按 `guize-buding-ku.md` 读取飞书 Base 中 `生效状态 = 已生效` 的规则补丁；读取失败不阻断审核，但必须写入风险说明。
4. 检查口径：先确认当前主销售口径和统计范围；费用率分母只在原报告出现费用率、ROI、预算费用或明确费用结论时审核。退款、净成交额、毛利、库存、复购只在原报告已提供或本月问题直接相关时审核，不作为常规必填项。
5. 读取正文和嵌入表格；如果有嵌入 Sheet，不得只读占位符，必须尽量读取 Sheet 数据。
6. 按 `shendu-shenhe-14-mokuai.md` 完成内部 14 模块审核。
7. 检查经营判断：问题归因、动作闭环、渠道/达人/客户、品项、费用效率、下月计划。
8. 按 `pingfen-guize.md` 评分，并应用最高分限制；如果命中已生效规则补丁，只在明确适用范围内覆盖内置规则。
9. 按 `shenhe-sop.md` 分流为 A/B/C/D。
10. 按 `shuchu-muban.md`、`zhuguan-zhenggai-bao.md` 和 `jiaofu-xingtai.md` 输出对应版本和可留档报告。

复审时额外执行：

1. 先按 `fushen-jianhua-rukou.md` 判断是否为简化复审入口。
2. 自动检索当前会话、当前工作区输出目录、用户提供链接及同部门同月份历史报告中的上次 AI 初审结果。
3. 找到上次初审依据后，直接对照整改后月报做复审。
4. 找不到上次初审依据时，只问用户补充“上次 AI 初审报告或主管整改版”，不要一开始要求全套材料包。

## 输出控制

正式初审的 14 模块审核和部长首屏是必交内容，不得被 A/B/C/D 分流结果省略。默认只生成一份完整 14 模块主报告 Word 正式版 + Markdown 底稿，避免部长打开多个文件；A 类额外单独生成老板一页纸正式版。B/C/D 的附加管理交付物默认合并到主报告前部，作为“部长重点看”章节，不另拆文件，除非用户明确要求单独文件。

- A：完整 14 模块主报告 + 部长首屏 14 模块速览 + 单独老板一页纸正式版。
- B：完整 14 模块主报告 + 部长首屏 14 模块速览 + 主报告前部“补充小问题清单 / 部长重点看”章节；不默认单独拆文件。

- 复审未通过：最多列 5 条剩余整改项。
- 简化复审入口：用户说“这是修改后的月报/整改后的月报/请复核”并发链接时，必须先自动找上次初审内容。能找到就直接复审；找不到再提示补充上次 AI 初审报告或主管整改版。
- 部长或管理者只贴链接发起正式初审时，必须生成完整 14 模块初审报告，默认同时生成 Word 正式版和 Markdown 底稿；聊天区最后必须附一段可直接转发给主管/部门的整改指令。
- 部长或管理者只贴链接发起正式初审时，聊天区必须按 `buchang-shouping-huifu.md` 输出“部长首屏回复”：读取状态、总结论、主要硬伤、14 模块速览、部长下一步、报告路径和可转发整改指令。不能只给 3 条问题和一个报告路径。
- 聊天区必须明确告诉部长“重点看主报告哪一节”。例如 B 看“补充小问题清单”，C 看“主管整改版 / 部长重点看”，D 看“重做通知 / 部长重点看”。A 类要同时给主报告路径和单独老板一页纸路径。
- 完整 14 模块初审报告前部必须说明报告生成逻辑，包括读取范围、审核路径、14 模块框架、评分权重、最高分限制和 A/B/C/D 分流规则。

每次输出前都要说明：

- 本次是否读取到飞书原文。
- 本次是否读取到关键表格。
- 本次是否读取到规则补丁库，以及命中了几条已生效规则补丁。
- 本次结论是正式审核、试审，还是材料缺失检查。

正式初审、深度审核、C/D 打回整改、复审结果留档，默认同时形成 Word 正式版和 Markdown 底稿。Markdown 用于追踪和二次编辑，Word 用于部长、主管、业务人员保存和流转。如果当前环境无法生成 Word，必须交付 Markdown 底稿并明确说明 Word 生成失败原因；不能只静默生成 `.md`。如果环境不可写，就在聊天区输出完整 Markdown 正文，并说明无法落盘生成 Word。

如果用户明确要求“完整审核、深度审核、14 模块、不要简单回答”，必须输出完整 14 模块报告，不能只输出 A/B/C/D 分流摘要。

### 首屏发送前强制自检

正式初审聊天回复发送前必须逐项检查：

1. 是否存在独立的 `1.` 到 `14.` 共 14 行；
2. 每行是否至少包含本模块判断、依据或材料状态、对部长的影响或下一步动作；
3. 是否把“14 模块速览”压成了一段由分号连接的总括句；
4. 是否把完整报告路径当成了 14 模块内容的替代；
5. 是否把“填写说明 / AI审核点 / 格式合规提醒”误列为错误。

第 1-4 项任意不满足时，回复不得发送，必须重新展开；第 5 项不满足时，必须删除该错误判断并恢复为“查看提醒，不计错”。

## 评分口径

满分 100 分：

| 维度 | 分值 |
|---|---:|
| 问题归因质量 | 30 |
| 数据真实性与一致性 | 25 |
| 执行质量与闭环 | 20 |
| 经营结果表现 | 15 |
| 下月计划可落地性 | 10 |

必须应用最高分限制。典型限制包括：

- 总销售额和分渠道合计差异超过 5% 且无解释：最高 60。
- 店铺销售和达人/KOL/KOC 销售混用且无说明：最高 65。
- 原报告已出现费用率、ROI、预算费用或明确费用结论，但费用率分母不清且影响核心费用结论：最高 80；多处分母冲突导致费用结论不可信时最高 70。
- 关键数据前后冲突：最高 70。
- 没有上月动作闭环：最高 80。
- 只有结果没有归因：最高 65。
- 明显模板错误、月份错误、字段未改：最高 80。
- 疑似重大数据造假或故意隐瞒：最高 50，并建议重做。

不得仅因未提供费用、退款金额、退款率、净成交额、毛利率、库存金额、库存状态、品项退款率或复购数据而扣分、封顶或打回。原报告未提供时可写“报告未提供，本次不作为基础审核必填项”；只有这些数据与本月明确问题、原报告结论或部长决策直接相关时，才提出轻量补充说明。

### 禁止默认整改项

正式初审和主管整改版不得把以下内容默认写成硬伤、必须补表或复审通过条件：品相分类、毛利档位、库存状态、退款/售后风险、复购、赠品成本、费用率复算表。只有原报告已经提供相关数据、已经提出相关结论，或本月问题直接指向这些项时，才允许作为条件补充。

## 风险边界

1. 报告未提供的数据必须写“报告未提供，无法核验”，不要补脑。
2. 口径差异不等于数据错误。先判断是确认错误、口径不清、数据缺失，还是正常业务差异但需要说明。
3. 业务结果差不等于月报质量差，业务结果好也不等于月报可以通过。
4. 测试模板、试填版、AI 试审版、模拟数据版不要写入真实部门月报共享目录。
5. 如果用户要求生成或上传测试材料，默认放到个人文档库或用户明确指定的私人测试位置。
6. 不输出、不记录任何密钥、Token、Cookie、数据库连接信息。
7. 规则补丁库只在公司飞书内使用，不生成公网公开链接，不使用个人或外包 profile。
8. 部长自然语言纠错满足规则补丁条件时，Codex 默认自动写入规则补丁库，不需要部长每次二次确认；不确定长期适用范围时写为 `待确认`。

## 给部长使用的建议话术

部长或其他同事可以直接发飞书链接，不需要记复杂话术：

```text
<飞书月报链接>
```

如果他们愿意加一句话，可以这样写：

```text
审核这个月报：<飞书链接>
```

复审时：

```text
这是修改后的月报，请做复核：<飞书链接>
```

如果 Codex 找不到上次初审内容，再补充上次 AI 初审报告或主管整改版即可。

如果部长发现审核结果不对，不需要填写反馈表，可以直接对 Codex 说：

```text
这里不对，达人组不需要销售回款，这个不应该扣分，以后遇到达人组按这个口径处理。
```

Codex 必须先修正当前报告，再按 `guize-buding-ku.md` 判断是否沉淀为规则补丁。涉及审核判断、扣分口径、小组类型、字段适用性、最高分限制或输出要求时，写入“销售月报审核规则补丁库”；只改措辞、格式、语气且不改变审核判断时，不写入。

满足规则补丁条件时，不需要再问部长是否写入；默认状态为 `待确认`。只有部长或刘涛明确表示长期生效，并且适用小组类型、适用场景、排除条件和评分影响都清楚时，才写为 `已生效`。写入前必须先查重，已有实质相同规则时不重复新增。
