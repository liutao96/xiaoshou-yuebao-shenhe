# 规则补丁库

## 定位

规则补丁库用于承接部长或管理者在使用 Codex 审核月报时的自然语言纠错。它不是公开资料库，也不是普通反馈表，而是销售月报审核 Skill 的受控规则补丁来源。

部长不需要手工填表，也不需要每次单独同意写入。部长已经有规则补丁库编辑权限时，部长在 Codex 里正常指出“这里不对”“这个不该扣分”“这个组不适用”“以后遇到这种情况按 XXX”，就视为授权 Codex 修正当前报告并沉淀规则补丁。

## 部长实际怎么用

部长使用 Skill 时不需要打开规则补丁库，也不需要填写反馈表。正常使用分三类：

1. 首次审核：部长直接把飞书月报链接发给 Codex，或说“审核这个月报：<飞书链接>”。
2. 复审：部长发“这是修改后的月报，请做复核：<飞书链接>”。Codex 先自动找上次初审报告，找不到时只让部长补上次 AI 初审报告或主管整改版。
3. 纠错：部长看到 Skill 判断不对时，直接对 Codex 说“这里不对，XX 组不需要 XX，这个不应该扣分，以后遇到这种情况按 XX 口径处理”。

纠错时 Codex 的顺序必须是：

1. 先修正当前报告或当前结论，不能只记录反馈。
2. 再判断这句话是否涉及审核规则、扣分口径、小组类型、字段适用性、最高分限制或输出要求。
3. 满足沉淀条件时写入规则补丁库；不满足时只修当前报告，不写入。
4. 写入后在聊天区说明“已修正当前报告，并沉淀 1 条规则补丁，状态为待确认/已生效”。

```powershell

  --base-token 'VKl3bMR5da6qKPsHU1Qc9UCYn94' `

  --filter-json '{"logic":"and","conditions":[["生效状态","==","已生效"]]}' `
  --limit 200 `
  --format json
```

如果当前 lark-cli 环境必须使用搜索接口，再使用带明确关键词的搜索，不要传空格 keyword：

```powershell
lark-cli base +record-search --profile company-liutao-default --as user `
  --base-token 'VKl3bMR5da6qKPsHU1Qc9UCYn94' `
  --table-id 'tbltqK12BlvyxE0n' `
  --json '{"keyword":"规则","search_fields":["规则标题","适用场景","部长修正口径"],"select_fields":["规则标题","规则类型","适用小组类型","适用场景","原错误判断","部长修正口径","评分影响","生效状态","规则优先级","生效说明"],"filter":{"logic":"and","conditions":[["生效状态","==","已生效"]]},"sort":[{"field":"更新时间","desc":true}],"limit":200}' `
```

读取失败时，不阻断单次审核，但必须在报告读取范围或风险说明中标注“本次未能读取规则补丁库，可能缺少部长后续修正口径”。

## 写入前去重

写入规则补丁前，必须先查重，避免同一类部长纠错被多个 Codex 会话反复写入。

查重优先按这些字段组合判断：

1. 规则标题相同或高度近似。
2. 适用小组类型相同。
3. 适用场景相同或高度近似。
4. 部长修正口径表达相同或实质相同。

推荐先按规则标题或关键词搜索已有记录：

```powershell
lark-cli base +record-search --profile company-liutao-default --as user `
  --base-token 'VKl3bMR5da6qKPsHU1Qc9UCYn94' `
  --table-id 'tbltqK12BlvyxE0n' `
  --json '{"keyword":"达人组销售回款","search_fields":["规则标题","适用场景","部长修正口径"],"limit":20}' `
  --format json
```

如果已存在实质相同规则：

- 不再新增重复记录。
- 如果原记录是 `待确认`，本次只能在聊天区说明“已存在相同待确认规则”，必要时补充来源链接或生效说明。
- 如果原记录是 `已生效`，直接按已生效规则修正当前报告，不再重复写入。
- 只有部长明确指出旧规则不完整、适用范围要扩大或状态要变更时，才进入更新已有记录流程；更新已有记录属于覆盖类操作，必须先询问刘涛或当前用户。

## 规则应用顺序

1. 先执行 Skill 内置规则，包括总控规则、14 模块、评分规则和最高分限制。
2. 再应用规则补丁库中 `已生效` 的命中规则。
3. 如果补丁规则与内置规则冲突，补丁规则只在其明确适用范围内覆盖内置规则。
4. 如果一条补丁的适用范围不清、状态不是 `已生效`，只能作为风险提示，不能改变评分。
5. 每次正式报告的“读取范围 / 生成逻辑”中，写明是否读取到规则补丁库，以及命中了几条已生效规则。

## 纠错沉淀规则

当部长或管理者在 Codex 中指出 Skill 判断偏差时，触发纠错沉淀。典型触发表达包括：

- 这里不对
- 这个不该扣分
- 这个组不适用
- 判断错了
- 以后遇到这种情况按 XXX
- 这个不是平台店铺组，是达人组 / 分销组 / 混合型
- 这个只能提示，不能作为打回条件

Codex 要先修正当前报告，再自动提炼一条规则补丁，写入 Base。不需要再问“是否写入规则库”，除非这次纠正涉及删除、公开权限、跨 profile、或会覆盖已有规则。

## 自动写入判定

部长自然语言纠错后，Codex 按以下规则自动判断写入方式：

| 场景 | 是否写入 | 生效状态 |
|---|---|---|
| 明确指出 Skill 判断错了、扣分错了、小组类型错了、字段不适用 | 写入 | 待确认 |
| 明确说“以后都按这个”“以后遇到这种情况按这个口径”且适用范围清楚 | 写入 | 已生效 |
| 明显字段适用性边界，例如某类小组不要求某字段，不属于单月特例 | 写入 | 已生效 |
| 只修改当前报告措辞、格式、语气，不涉及审核规则 | 不写入 | 不适用 |
| 只要求当前报告补充解释、换个说法、删掉一段话，但没有改变审核判断 | 不写入 | 不适用 |
| 只针对当月特殊情况，不适合长期复用 | 写入 | 待确认 |
| 无法判断是规则问题还是当前报告个例 | 写入 | 待确认，并在生效说明里写“需人工确认适用范围” |
| 涉及公网开放、跨 profile、删除记录、覆盖已有规则 | 暂停并询问刘涛或当前用户 | 不适用 |

原则：先修当前报告；只有涉及审核规则才沉淀；能明确长期复用才 `已生效`；不确定也写入，但保持 `待确认`；纯表达、格式和语气调整不进入规则补丁库。

默认写入规则：

| 字段 | 默认值 |
|---|---|
| 生效状态 | 待确认 |
| 规则优先级 | 普通 |
| 纠错来源 | 部长Codex自然语言 |
| 修正人 | 当前飞书 user，能识别则写入，不能识别则留空 |

只有以下情况可以直接写为 `已生效`：

1. 刘涛明确说“这个规则以后按这个执行”。
2. 部长明确说“以后都按这个规则”，且适用范围清楚。
3. 这是明显字段适用性边界，例如某类小组不适用某个字段，不属于单月特例。
4. 已经明确写出适用小组类型、适用场景、排除条件和评分影响；任一项不清楚时，先写 `待确认`。

不满足以上条件时，先写 `待确认`，不能直接污染正式规则。

如果部长已经被加入 Base 协作者并具备编辑权限，部长 Codex 写入规则补丁库属于本 Skill 的默认动作，不需要每次弹出二次确认。写入后要在聊天区简短说明：已修正当前报告，并已沉淀 1 条规则补丁，状态为 `待确认` 或 `已生效`。

## 权限边界

1. 不创建公开分享链接。
2. 不把 Base 改成互联网可访问。
3. 不使用 `personal-do-not-use-company-content` 或 `outsource-approval-report-sync-bot-no-user`。
4. 不把 token、cookie、secret 或 appSecret 写入记录。
5. 普通组长默认不需要看到规则补丁库；需要查看时按公司飞书权限单独授权。
6. 规则补丁库只记录纠错规则，不保存完整月报正文或敏感经营明细。
